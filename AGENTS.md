# MyBook 持续性指令

本文件只记录用户在本项目对话中明确给出的长期指令。若用户之后明确覆盖，以之后的指令为准，并更新该文件。

## 操作约定

- 如果用户只要求方案、解释，或明确说“先不要修改代码”，只给出方案或解释，不实现。
- `AGENTS.md` 只需要在对话开始时加载；除非用户明确要求重新加载，或本文件被修改，否则同一对话中不需要每个请求或每条命令前重新读取。
- 当用户只发送 `p` 时，将代码修改提交并推送；只提交代码，不提交临时文件、配置文件、二进制文件、本地报表、下载的账单等。
- 当用户发送以 `todo` 开头的消息时，不执行 `todo` 后面的指令，而是将其作为一条新的待办追加写入项目根目录的 `TODO.md`；如果 `TODO.md` 不存在则创建。
- 当用户只发送 `do` 时，从项目根目录的 `TODO.md` 读取最新一条待办并告诉用户，但不要执行该待办；随后将这条待办从 `TODO.md` 删除。
- 生成 commit message 时必须写明这是由 AI 生成的。
- 当用户只发送 `c` 时，清理数据库中固定数据以外的数据；固定数据包括 `Account` 和 fake/checkpoint `StatementImport`。
- 当用户只发送 `t` 时，清理数据库并测试 `FetchICBCBills`。
- 当用户只发送 `r` 时，清理数据库并重新执行一轮测试。
- 如果需要运行 `MyBook.exe` 时发现程序已经在运行，先向用户获取关闭程序的许可。

## 数据准确性与 Record

- 不管哪个模块哪个功能，所有数据以精确为第一目标，包括 `Record` 和 `Balance`。
- 永远不允许擅自生成虚构的 `Record`；只有用户可以手动增加没有来源的 `Record`。
- 不管哪个模块，除非用户明确指示，不允许擅自抵消、合并、忽略 `Record`。
- 所有 `Record` 相关导入，只要能读分项就不要读总项；`Record` 总是应该尽可能细致。
- 总项、期初余额、期末余额等只能用于校验，不能替代明细生成 `Record`。
- 不允许忽略 1 分钱或其它小额差异；出现差异必须查找来源。
- 不允许任何残差；不得通过生成“残差”“精度差异”“口径差异”等 `Record` 让账单或报表校验通过，必须查明来源并修复解析逻辑或拒绝导入。
- `Record` 必须关联到一个 `StatementImport`。
- 来自同一个邮件、账单、报表或同一次操作的所有数据库修改，必须作为一次原子操作写入数据库，保证数据库总是自洽。
- 从外部获取信息时，`Record` 和 `Balance` 总是应该同步修改。只有手动修改时允许 `Record` 和 `Balance` 不同步，但必须弹窗提示确认。
- `relativeBalance` 账户的初始余额为 0；写入或删除 `Record` 时同步修改余额。如有需要，由用户用手动校正 `Record` 修正推算余额和真实余额的差异。
- 内部交易表示用户自己的账户间互转，不影响总资产；即使是内部交易，它产生的手续费、佣金等也会造成总资产损失，应该对应单独的非内部 `Record`。
- 所有显示到 UI 上的数额，除非特别说明，否则都不计算内部转账和已抵消的退款。
- 跨月互相抵消的记录不能删除，以防造成期初快照加上 `Record` 之和不等于期末快照。
- `isRefundMatched` 表示退款和对应消费已经匹配；这些记录默认不计入界面统计图表，但其它情况下仍正常参与计算。
- `Reason` 里不要体现 Wise、IBKR 等来源名；这些信息和 account id 提供的信息重复。

## 金额、币种与持仓

- 所有涉及金钱、价格的变量都要注意币种，并尽量以 `Currency` 而非数值作单位。
- 计算现金或资产总价时只精确到两位小数；资产单价允许使用其它精度，解析报表时使用合适精度。
- 数据库存储货币类型时使用枚举而非整型，以增强可读性。
- 今后增加的所有枚举类型都要尽可能存储为 MySQL `ENUM`。
- 数据库里的 `ENUM` 不能出现中文。
- 由于 HeidiSQL 有 bug，不能正确区分 `Unknown` 类型，今后数据库 `ENUM` 中要避免使用 `Unknown` 作为枚举值。

## 数据库与实体

- 所有代码中赋予默认值并需要持久化的成员，其默认值也应作为数据库列默认值。
- 目前和今后新增的所有关联关系都使用外键。
- 当修改代码中的数据结构时，除非用户说要兼容，否则直接执行数据库修改；不要等待下次运行时才修改数据库，也不必保留修改数据库的代码。
- `StatementImportStartDate` 不要体现在代码里，通过向数据库写入 fake 数据实现。
- 数据库中的 `Account` 和 fake/checkpoint `StatementImport` 是固定数据，永远不要修改，除非用户明确要求。
- `Account` 是不常增加的数据，不允许导入流程直接创建；所有查找 `Account` 的地方只允许 Get，找不到就报错。
- 数据库中的枚举值不要使用中文。
- 对于现有和新增的模块，除非用户明确说明要读取 `Account.email` 或邮箱属性，否则忽略邮箱属性。
- 账号或卡号只允许从总结、摘要中明确写明为账号、卡号或类似含义的数据读取；不要从交易明细中推断或登记账号、卡号。
- IBKR 报表中的账户号也视为账号或卡号信息。
- 除非用户特别说明，否则卡号表只增不减，不自动删除已有卡号记录。

## 邮件、网络与定时任务

- 获取邮件或附件总是直连，不需要配置项，不受系统代理影响。
- Debug 调试时不执行定期 fetch 任务。

## UI

- 所有数值都应可以通过鼠标选中并通过 Ctrl+C 复制。
